    Using this system prompt

    What It Covers
    Core Accessibility Requirements

    The system prompt enforces non-negotiable accessibility standards:

    • WCAG 2.2 AA Compliance - All generated code must meet Level AA success criteria including colour contrast ratios, keyboard navigation, focus indicators, and semantic structure
    • The Five Rules of ARIA - Comprehensive guidance on when and how to use ARIA, with emphasis on using native HTML first
    • Focus Indicators - Detailed patterns for keyboard focus that work across all backgrounds and in Windows High Contrast Mode
    • Motion Preferences - Mandatory prefers-reduced-motion support for all animations
    • Screen Reader Considerations - Proper labelling, semantic structure, and announcement behaviour
    Colour and Contrast Management

    The prompt includes an extensively tested colour palette with pre-validated pairings:

    • Pre-tested Palette - Semantic CSS custom properties that automatically switch between light and dark modes
    • Safe Pairings Matrix - Explicit documentation of which colour combinations pass WCAG 2.2 AA in both light and dark modes
    • Anti-patterns - Clear examples of what not to do, as 51-58% of possible colour combinations fail accessibility requirements
    • Decision Trees - Step-by-step guidance for selecting appropriate colour pairings

    The accompanying accessible-palette.css file provides 12 semantic colour variables for backgrounds, text, accents, borders, and state indicators.

    Critical Usage Guidelines

    The prompt prevents common accessibility mistakes:

    • Never Use Title Attributes - Comprehensive explanation of why title creates barriers and what to use instead
    • Aria-Label Limitations - When to use it, when not to use it, and its limitations across assistive technologies
    • Windows High Contrast Mode - Specific patterns for supporting users who override author styles
    • Icon Handling - Always hide decorative icons with aria-hidden="true" and provide visible text
    Technology Stack and Constraints

    The prompt enforces a specific, accessible technology stack:

    • Semantic HTML5 - Native elements first, custom widgets only when necessary
    • CSS - Modern CSS including custom properties, no preprocessors
    • Vanilla JavaScript - No frameworks, no build tools, no npm
    • CDN-Only Dependencies - External libraries must be loaded from CDN
    • HTML-First Development - Write HTML directly in files; avoid generating HTML with JavaScript
    File Management and Workflow

    Critical protocols to prevent wasted effort and broken code:

    • Mandatory Current File Requests - LLM must always ask for current file contents before modifying
    • One File at a Time - Work on single files with testing verification before proceeding
    • Incremental Development - Step-by-step modifications with console testing between each step
    • Integration Checks - Verify import/export patterns, method naming, and dependencies before changes
    Code Quality Standards

    Consistency and maintainability requirements:

    • British English - All comments, variables, and documentation use British spelling
    • Configurable Logging - Standard logging configuration for all JavaScript files with ERROR, WARN, INFO, and DEBUG levels
    • Clear Instructions - Step-by-step modification format with line numbers and complete code blocks
    Project Viability Assessment

    Guidance on which projects are suitable for vibe-coding:

    • Ideal Projects - Client-side only applications with no server requirements
    • Acceptable with Caution - Projects using public APIs with CORS, GDPR, and rate limiting considerations
    • Never Attempt - Projects requiring authentication, databases, payment processing, or storing personal data
    How to Use It
    Setting Up the System Prompt

    The system prompt is designed to be used with Claude or other LLMs that support system prompts or custom instructions:

    1. For Claude.ai - Add the prompt content to your Project Knowledge or Custom Instructions
    2. For Claude API - Include the prompt in the system parameter of your messages
    3. For Other LLMs - Add as custom instructions, system prompt, or initial context depending on the platform
    Including the Colour Palette

    The accessible-palette.css file should be:

    1. Uploaded to your project as a reference file
    2. Linked in your HTML: <link rel="stylesheet" href="accessible-palette.css">
    3. Referenced in conversations when discussing colour choices

    The LLM will use the documented colour variables and never suggest untested colour combinations.

    Starting a New Project

    When beginning a new accessible web project:

    1. Verify your project is suitable using the viability checklist
    2. Describe your project goals and requirements
    3. The LLM will use semantic HTML, the accessible colour palette, and vanilla JavaScript
    4. Request one feature or file at a time
    5. Test each component before requesting the next
    Modifying Existing Code

    When working with existing files:

    1. Share the current file contents with the LLM
    2. Describe the changes you need
    3. The LLM will provide step-by-step modification instructions
    4. Implement one step at a time
    5. Test in the browser console after each step
    6. Report any errors before proceeding
    Expected Interaction Pattern

    The system prompt creates this workflow:

    1. Human requests a feature - Describe what you want to build
    2. LLM asks for current file - Never assumes it has the latest version
    3. Human provides file contents - Copy and paste current code
    4. LLM provides numbered steps - Clear find/replace instructions
    5. Human implements changes - Update code one step at a time
    6. Human tests in browser - Check console, keyboard navigation, screen reader
    7. Human reports results - Confirm success or describe errors
    8. Repeat for next file - Only proceed after current file works
    Key Features
    Accessibility-First Decision Making

    Every suggestion prioritises removing barriers:

    • Native HTML elements over custom widgets
    • Visible text over icon-only interfaces
    • Keyboard accessibility by default
    • Screen reader compatibility in all interactions
    Prevention of Common Mistakes

    The prompt actively prevents problematic patterns:

    • Using title attributes for important information
    • Overusing aria-label when visible text would work
    • Creating colour combinations that fail contrast requirements
    • Building custom controls without keyboard support
    • Hiding focus indicators for mouse users
    Token Awareness

    The prompt makes the LLM conscious of usage costs:

    • Ask clarifying questions before generating large amounts of code
    • Request current file contents rather than guessing
    • Provide correct solutions on the first attempt
    • Avoid creating bugs that waste tokens fixing
    Educational Component

    While generating code, the LLM explains:

    • Why specific accessibility patterns are used
    • What problems are being prevented
    • How the code supports different assistive technologies
    • WCAG success criteria being met
    When to Use This System Prompt
    Ideal Use Cases

    This system prompt excels for:

    • Learning accessible development - Students and developers new to accessibility
    • Rapid prototyping - Creating accessible proof-of-concepts quickly
    • Client-side tools - Calculators, converters, validators, visualisations
    • Educational resources - Building inclusive learning materials
    • Portfolio sites - Creating accessible personal or professional websites
    • Accessibility auditing tools - Building tools to test other sites
    When to Consider Alternatives

    This approach is not suitable for:

    • Projects requiring user authentication
    • Database-backed applications
    • Payment processing systems
    • Projects collecting personal data
    • Real-time collaboration requiring servers
    • Large-scale production applications requiring frameworks

    For these projects, professional development practices with proper security, testing, and infrastructure are essential.

    Getting Started
    Prerequisites

    Before using this system prompt, you should have:

    • Basic understanding of HTML, CSS, and JavaScript
    • A text editor for writing code
    • A modern web browser for testing
    • Access to Claude.ai or another LLM that supports system prompts

    You do not need to be an accessibility expert; the system prompt provides that expertise.

    Your First Project

    Try building something simple to learn the workflow:

    1. Choose a simple project - A colour contrast checker or accessible form validator
    2. Set up the system prompt - Add it to Claude's Project Knowledge
    3. Upload accessible-palette.css - Include it as a reference file
    4. Describe your project - Explain what you want to build
    5. Follow the step-by-step instructions - Implement one change at a time
    6. Test thoroughly - Check keyboard navigation, colour contrast, and screen reader behaviour
    Testing Your Code

    After implementing changes, always test:

    1. Browser console - Check for JavaScript errors (should be zero)
    2. Keyboard navigation - Tab through all interactive elements
    3. Focus indicators - Verify visible focus states on all interactive elements
    4. Colour contrast - Use browser DevTools or online checkers
    5. Screen reader - Test with NVDA (Windows), JAWS (Windows), or VoiceOver (macOS/iOS)
    6. Zoom - Test at 200% zoom level
    7. Windows High Contrast Mode - Verify visibility of focus states and borders
    Best Practices
    Communication with the LLM
    • Be specific - Describe exactly what you want to achieve
    • Share complete files - Always provide current file contents when modifying existing code
    • Report test results - Let the LLM know what works and what doesn't
    • Ask questions - If you don't understand why something is done a certain way, ask
    Incremental Development
    • One file at a time - Complete and test each file before moving to the next
    • Small steps - Request small changes rather than large rewrites
    • Test frequently - Check the browser console after every change
    • Version control - Keep backup copies of working code
    Learning and Understanding
    • Read the explanations - The LLM explains why certain patterns are used
    • Refer to the system prompt - Consult the colour palette guide and ARIA rules
    • Build progressively - Start with simple projects and increase complexity
    • Contribute back - Share what you learn with others building accessible sites
    Conclusion

    This system prompt transforms LLMs into accessible-first development assistants that help you build inclusive websites. By enforcing WCAG 2.2 AA standards, preventing common mistakes, and providing clear guidance, it enables developers of all skill levels to create digital experiences that work for everyone.

    Remember: accessibility is not optional. Every barrier you remove makes the web more usable for disabled people, and often improves the experience for everyone.
